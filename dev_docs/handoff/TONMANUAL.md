# Squid-N トンマナガイド (TONMANUAL)

Squid-N のトーン＆マナー（デザイン／ライティング規約）を定義する。本書は **UI／配色／文言の一貫性を保つための共有仕様** であり、特定の画面実装に依存しない値・原則のみを記載する。同じ製品グループの関連プロダクトとトーンを揃える起点でもある。

- プロダクト性格: 日本の建築構造計算一貫プログラム。**Rust + egui/eframe** で作られたデスクトップアプリ（`squid-n-app`）。モデル作成 → 荷重 → 解析（静的／固有値／地震／プッシュオーバー／時刻歴）→ 検定・設計 → レポートまでを 1 本で完結させる。
- 設計原則: **明るく・クリーンで・データ（モデルと解析結果）を主役にする**。UI クロームは控えめなブルー基調、コンテンツ（3D ビュー・グラフ・テーブル）は白背景で最大限のコントラストを確保する。

> 実装上の単一情報源は `crates/squid-n-app/src/theme.rs`。本書 §2／§3／§3-2／§4／§6 の値はそのまま同ファイルの定数・`apply_theme` に反映されている。色や寸法を変えるときは本書と `theme.rs` を同時に更新する。

---

## §1. デザイン原則

1. **データファースト** — UI 要素はモデル・結果の邪魔をしない。3D ビュー／グラフは白背景 (`#FFFFFF`)、装飾は最小限。
2. **静かなクローム** — ツールバーやパネルは淡いブルー／グレーで沈ませ、ユーザーの注意をコンテンツへ向ける。
3. **一貫したブルー** — ブランドのアクセントは blue-500 (`#3B82F6`)。インタラクティブ要素・選択・リンクはこのブルー系で統一する。
4. **意味のある色** — グラフ・図の色は装飾ではなく状態・カテゴリを表す（検定 OK＝緑、超過＝赤 など）。
5. **ライトテーマ基準** — ライトテーマを正とする。`apply_theme` は eframe がダーク起動する場合も先に `Visuals::light()` で上書きする。

### 配色の原則

- **背景の階層**: 白（コンテンツ） < gray-100（パネル／ツール領域） < blue-200（ナビゲーション）。手前のクロームほど青く、奥のコンテンツほど白い。
- **テキスト on ブルー背景**: ホバー／アクティブ時は白。通常時は gray-700。
- **コントラスト確保**: 選択ハイライトは背景 blue-200 より一段濃い blue-300 を使い、状態を視認可能にする（同系色の潰れを避ける）。

## §3. データビジュアライゼーション配色

3D モデル・変形図・応力図・グラフ（プッシュオーバー性能曲線・時刻歴・MN 相関曲面）・検定結果の色は、装飾ではなく **意味（状態・カテゴリ）** を担う。基調は以下の 5 色（+ 二次部材の暖色）。

| カテゴリ | 色 | HEX | 意味（構造解析での用途） |
| --- | --- | --- | --- |
| 標準・既定 | 青 | `#4285F4` | 既定の線・要素・変位、グラフの基準系列 |
| 超過・危険側 | 赤 | `#EA4335` | 検定比 > 1.0（NG）、応力集中、負方向の量、崩壊ヒンジ |
| 注意・中間 | 黄 | `#FBBC04` | 検定比 0.8〜1.0、中間状態、予測値 |
| 良好・OK | 緑 | `#34A853` | 検定比 ≤ 0.8（OK）、収束、正常状態 |
| ハイライト | 紫 | `#7C4DFF` | 選択中の節点・部材・断面のハイライト |
| 二次部材 | アンバー | `#D97706` | 小梁・間柱など**解析対象外の実在部材**の線・輪郭（黄と同族の暖色、線の視認性のため濃いめ） |

### 検定比→状態色の写像

検定比などの「余裕度」は上記 3 色へ機械的に写像する（`theme.rs::status_color`）。

- `ratio ≤ 0.8` → 緑（良好）
- `0.8 < ratio ≤ 1.0` → 黄（注意）
- `ratio > 1.0` → 赤（超過・NG）

### グレーアウト規約

- 選択／フィルタ外の点・線・部材は **色相を捨てて半透明グレー** にする。選択中の対象を引き立て、元のカテゴリ色と混同させない。
- 応力図・面要素の塗りつぶしは `theme.rs::translucent` で半透明化し、背後のモデル線を透かす。

### カラーマップ（連続値）

- 応力コンター等の連続値表示は、知覚均等で色覚多様性に配慮した **Viridis を既定**とする。

## §3-2. 3D ビュー規約

3D モデルビューア（架構モデル・変形図・応力図・崩壊機構）は配色・カメラ・操作を全ビューで統一する。

### 背景・グリッド・軸の配色

| 要素 | HEX / 値 | 備考 |
| --- | --- | --- |
| 3D 背景 | `#F0F2F5` | **2D グラフの白とは異なり淡いグレー**。立体感と奥行きを出すため意図的に白を避ける |
| グリッド線 | ダーク半透明・線幅 0.5 | XY/XZ/YZ の 3 面、各軸 4 分割 |
| X 軸 | `#D26464`（赤系・線幅 1.5） | |
| Y 軸 | `#50AA50`（緑系・線幅 1.5） | Y＝奥行き |
| Z 軸 | `#6464C8`（青系・線幅 1.5） | Z＝鉛直（重力方向） |

> 3D 軸色は彩度を落とした専用色で、§3 のグラフ 4 色（青/赤/黄/緑）とは別系統。**赤=X / 緑=Y / 青=Z** の対応を全 3D ビューで固定する。ビューキューブ（`viewer/viewcube.rs`）は Front をワールド -Y 側とする（Y=奥行き・Z=鉛直の慣例）。

### 軸ラベル

- 正方向の端: `軸名 (最大値)` を 11px、軸色で表示。
- 負方向の端: `最小値` を 10px、軸色の淡色（約 70% 明度、`theme.rs::lighten`）で表示。
- データ範囲 `[min, max]` は常にクリップ空間 `[-1, 1]` に正規化してから描画する。範囲ゼロの軸は中央 0 に潰す。

### カメラ

| 項目 | 既定値 | 範囲 |
| --- | --- | --- |
| 回転 | 正面 | — |
| ズーム | 3.0 | 0.5 – 10.0 |
| パン | (0, 0) | — |

- **ターンテーブル（クォータニオン）回転**を採用する。水平ドラッグ＝鉛直軸（Z）まわりの旋回、
  垂直ドラッグ＝俯仰（真上〜真下でクランプ）。ロールが発生しないため鉛直軸は常に画面上で縦に保たれる
  （自由回転のアークボールはロールが蓄積し視点が傾くため採らない。建築モデルは鉛直の維持が重要）。
- 投影は正射影ベース。

### インタラクション規約

| 操作 | 動作 | パラメータ |
| --- | --- | --- |
| 左ドラッグ | 回転 | 感度 0.005 /px |
| スクロール | ズーム | 係数 0.01、0.5–10.0 にクランプ |

全 3D ビューでこの操作割り当てを統一する（マウス操作の意味をビュー間で変えない）。ビューキューブの面クリックで正投影の代表視点へスナップする。

### 深度・重なり

- サーフェス（応力図・面要素）や軸を線・点と正しく前後表示する場合は、軸を細分化して同じ深度ソートに混ぜる。

## §4. タイポグラフィ

`theme.rs::apply_theme` が egui の `TextStyle` に以下を割り当てる。

| TextStyle | サイズ(px) | ファミリ | 用途 |
| --- | --- | --- | --- |
| Heading | 18 | Proportional | 見出し |
| Body | 13 | Proportional | 本文・ボタン基準 |
| Button | 13 | Proportional | ボタンラベル |
| Monospace | 12 | Monospace | 数値・コード表示 |
| Small | 11 | Proportional | セル内・補助ラベル |

新規 UI は **11 / 12 / 13 / 見出し(18)** のスケールに収め、独自サイズの乱立を避ける。

### 導出原則: 文字サイズは意味で指定し、寸法は導出する

文字サイズに関わる値を「意味付きスタイル」と「寸法の固定 px」の二重管理にしない。

- **フォントサイズそのものは `TextStyle`（Heading/Body/Button/Monospace/Small）でのみ指定する**。
  ウィジェットのテキストに `FontId::new(13.0, …)` のような生の pt 指定を書かない。
  サイズを変えたい場合は `apply_theme`（1 箇所）を直す。
- **テキストを内包する箱の寸法**（テーブル行高・ヘッダ高・ステータスバー高など）を
  固定 px で書かない。フォント（特に和文フォールバック）の行高は欧文フォントの想定と
  ずれるため、固定値は環境・フォント変更のたびに文字が見切れる形で壊れる。必ず
  `ui.text_style_height()`（例: `theme.rs::table_row_height`）から導出し、余白量も
  名前付き定数として理由をコメントする。
- **固定 px が許されるのは非テキスト形状のみ**（アイコン半径・線幅・3D ガジェットの
  スケール・当たり判定半径など）。文字が入らない幾何は固定でよい。

## §5. レイアウト & スペーシング

### 構造

`squid-n-app` は egui の `SidePanel`／`TopBottomPanel`／`CentralPanel` で以下を構成する。

```
┌─────────────────────────────────────────────────────┐
│ 工程タブ（モデル / 荷重 / 解析 / 結果 / 設計）        │  上部：作業フローに沿った工程切替
├──────────┬──────────────────────────────┬───────────┤
│ ナビゲータ │   中央：3D ビューア（white）   │ インスペクタ │  中央はモデル/変形図/応力図
│ (gray)   │   / テーブル / グラフ         │  (gray)   │  左右クロームは gray-100
│ 階·部材群 │                              │ 選択対象  │
│ 荷重·結果 │                              │ の詳細    │
└──────────┴──────────────────────────────┴───────────┘
```

- **工程タブ**（モデル → 荷重 → 解析 → 結果 → 設計）は作業フローの順序を体現する。左から右へ進むほど下流工程。
- **ナビゲータ（左）**: 階／部材グループ／荷重ケース／結果ケースのツリー。項目クリックで中央ビュー・テーブル・インスペクタが連動する（双方向連動）。
- **中央キャンバス**は常に白で、モデル・変形図・応力図・グラフを表示する。
- **インスペクタ（右）**: 選択中の節点・部材・断面の詳細と編集導線。

### スペーシング指針

- パネル内側余白は 8px を基準。標準の項目間隔を基本とし、密集領域（リスト・アイコン列）のみ明示的に縮める。
- 中央コンテンツ幅を優先し、クロームは控えめに。モデル編集は常に中央ビューへ即時反映する（stale 表示を残さない）。

## §6. コンポーネント規約

`theme.rs::apply_theme` の `Visuals` 設定に対応する。

### 角丸（Corner Radius）

| 要素 | 半径 |
| --- | --- |
| ボタン・入力欄・コンボボックス | 4px |
| カード／パネル／ウィンドウ／メニュー | 6px |

→ **小要素=4px、カード=6px** を基準にする（`theme.rs` の `r4` / `r6`）。

### 影（Elevation）

- フローティング UI（ウィンドウ・メニュー・一時パネル）のみ影を持つ。中央コンテンツやセルはフラット（影なし）。

### ボタン（ウィジェット状態）

- **通常 (inactive)**: 透明背景・gray-200 枠・gray-700 テキスト。
- **ホバー (hovered)**: 背景 blue-300・白テキスト。
- **アクティブ／選択 (active)**: 背景 blue-500・白テキスト。
- **無効**: テキストを淡色化、透明背景。

### 入力欄・コンボボックス

- 背景 gray-100、枠線 gray-200、テキスト gray-700。

### テーブル

- 行ストライプは gray-200（白背景上で淡すぎる色は避け、視認できる濃さにする）。
- 行高は `table_row_height`（Body 行高＋余白）で導出する（§4）。

### パネル背景

- 左右パネル・工程タブ = gray-100。中央ビュー本体 = 白。

## §7. アイコン & インジケーター

- システム絵文字を機能アイコンとして使用（例: `📥 ST-Bridge 読込`、`📤 書出`、`✏ 適用`、`🏢 自動生成`、`▶ 解析実行`、`▶▶ 一括解析`）。意味が直感的な絵文字に限り、装飾目的では使わない。
- 実行系（解析・生成）はテキストの先頭に動作を表す絵文字を置き、危険・注意は `⚠` を用いる。
- ステータスバー・ツールチップ（`on_hover_text`）で操作の前提や意味を補足する（例: 「固有値解析による 1 次周期」）。

---

## §8. モーション

- **控えめ・短い**: アニメーションは状態遷移の理解を助ける範囲に限る。アニメーション中のみ再描画し、定常状態では無駄に再描画しない。
- 画像／PNG キャプチャ中は装飾インジケーターを描画しない（成果物への写り込み防止）。

## §9. ボイス & トーン（ライティング）

### 言語ポリシー

- **アプリ UI ラベル・メッセージは日本語**（例: 「解析設定」「荷重組合せ」「検定結果数」）。Squid-N は日本での利用を想定し、UI・エラー・ツールチップとも日本語で統一する。
- **例外（慣例で英語）**: 3D ビューキューブの面ラベル（Front / Back / Top / Bottom / Left / Right）、軸記号（X / Y / Z）、単位・記号（mm、kN、N/mm²、Ds、Rt、Ai など）。これらは構造・CAD の国際慣例に従う。
- 計算根拠ドキュメント（`docs/`）は利用者向けの日本語で記述する（英語版は用意しない）。

### 文体

- UI ラベルは **簡潔・体言止め／短い動詞句**。冗長語を避ける（「読込」「全組合せ一括解析」）。
- ヘルプ・ツールチップは丁寧かつ実務的。専門用語（許容応力度・保有水平耐力・偏心率・限界耐力）は読者を構造設計実務者と想定し、過度に噛み砕かない。
- エラーは事実ベースで簡潔に。**原因と次の操作が分かる表現**を優先する（例: 「荷重ケースがありません。荷重タブで作成してください。」「モデルが編集されました。結果は再計算が必要です。」）。

## §10. 統一チェックリスト

新しい UI／図表を追加・移植するときは以下を確認する。

- [ ] 色は本書のパレット（§2・§3）＝ `theme.rs` の定数を使ったか（独自色を散らさない）
- [ ] 背景階層（白＜gray-100＜blue-200）に沿っているか
- [ ] フォントサイズは `TextStyle` スケール（11/12/13/見出し）内か（生の `FontId` pt を書いていないか）
- [ ] テキストを含む領域の高さを固定 px で書いていないか（`text_style_height()` から導出したか）
- [ ] 角丸は 4px（小要素）/ 6px（カード）か
- [ ] 影はフローティング UI のみか（コンテンツはフラット）
- [ ] 図・グラフの色が状態・カテゴリの意味と一致しているか（検定 OK=緑／注意=黄／超過=赤）／選択外はグレーアウトか
- [ ] 3D ビューは背景グレー・赤=X/緑=Y/青=Z・ターンテーブル操作に揃っているか
- [ ] UI ラベル・メッセージは日本語・簡潔か（慣例英語の例外のみ許容）
- [ ] 連続値カラーマップの既定は Viridis か
